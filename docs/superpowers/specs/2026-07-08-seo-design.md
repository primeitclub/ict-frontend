# SEO Improvements — Design

## Context

`ict-frontend` is a client-side-rendered (CSR) React + Vite SPA with no head-management
library, no meta description/OG tags beyond a single static `<title>`, no `robots.txt`
or `sitemap.xml`, and no structured data. Production deploys are FTP'd static files to
`ftp.primeitclub.com` — there is no confirmed server-side scripting available at request
time, so any per-page dynamic rendering has to happen client-side after the JS bundle
loads.

Routing is version-scoped (`/v5`–`/v8`, latest also mounted at root) via React Router v7,
with a dynamic `event-detail/:eventId` route. `useEventDetail` already fetches everything
needed for per-event metadata (title, description, image, date, location, speaker).

**Known limitation accepted by this design:** because we are staying CSR-only, non-JS
crawlers (WhatsApp, Twitter/X, LinkedIn, Facebook link previews, and some search bots)
will only ever see the sitewide default OG tags baked into `index.html` — not per-event
titles/images. Only JS-executing crawlers (notably Googlebot) will see the dynamic,
per-route metadata described below. Fixing per-event social previews for non-JS crawlers
would require server-rendered or prerendered HTML, which was explicitly descoped due to
uncertainty about the static host's support for request-time scripting or routing rewrites.

## Goals

- Improve search discoverability for JS-executing crawlers.
- Ship a solid, correct default social preview card sitewide.
- Establish baseline technical SEO hygiene (robots.txt, sitemap.xml, canonical URLs).
- Add structured data for events/organization where feasible without a server.

## Scope

Current/latest flagship version only. Historical versions (v5, v6, ...) are out of scope —
they remain CSR with no additional SEO treatment.

## Design

### 1. Sitewide baseline (`index.html`, `public/`)

- `index.html`: add meta description, default Open Graph tags (`og:title`,
  `og:description`, `og:image`, `og:type=website`, `og:site_name`), and Twitter Card
  tags (`summary_large_image`). These are the fallback for any route without
  page-specific tags, and are what non-JS crawlers see everywhere.
- `public/robots.txt` (new): allow everything except `/admin/*`; reference the sitemap.
- `sitemap.xml`: generated by a small Node script run as part of `yarn build`. Includes
  static routes for the current version (home, events, sponsors, teams, register,
  contacts) plus one `<url>` entry per current-version event (from the event API),
  with `lastmod` from each event's `updatedAt`. Written to `dist/sitemap.xml` post-build.

### 2. Dynamic per-route head management

- Add `react-helmet-async`. Wrap the app in a single `<HelmetProvider>` near the root.
- Add a small reusable `<Seo>` component (props: title, description, canonical URL,
  optional OG overrides) used on each route:
  - Home, events list, sponsors, teams, register, contacts: static per-route title/
    description strings, each with its own canonical URL.
  - Event-detail: dynamic title (`"{event.title} | Prime ICT Meetup"`), description
    from `event.description` (truncated to ~155 chars), canonical URL for that event,
    OG image = `event.imageUrl` falling back to the sitewide default.
- This fixes browser tab titles and gives Googlebot (which executes JS) correct
  per-page titles/descriptions on its render pass. It does not change what non-JS
  social crawlers see (see Known Limitation above).

### 3. JSON-LD structured data (client-side only)

Delivered via the same `react-helmet-async` mechanism, no static baking:

- **Organization** schema (name, logo, URL, social links) on the home page.
- **Event** schema on event-detail pages (name, startDate, location, description,
  image, organizer).

Benefits JS-executing crawlers only (primarily Googlebot); near-zero marginal cost
since the underlying data is already fetched by existing hooks.

## Explicitly out of scope (descoped during brainstorming)

- Build-time/scheduled static prerendering of event-detail pages for accurate
  per-event social preview cards on non-JS crawlers. Descoped due to uncertainty
  about whether the static FTP host supports request-time scripting or the routing
  rewrites needed to serve prerendered files at `/event-detail/{id}`.
- Full SSR migration (e.g. moving off Vite CSR to a server-rendered framework).
- SEO treatment for historical versions (v5–v7 if v8 is current).
- Dedicated speaker/sponsor detail pages (none currently exist as routable entities).

## Testing / verification

- Manual: inspect rendered `<head>` via browser devtools on each route after
  client-side navigation; confirm helmet updates title/meta/canonical correctly.
- Validate `sitemap.xml` output against the sitemap XML schema and confirm it only
  lists current-version URLs.
- Validate JSON-LD blocks with Google's Rich Results Test / Schema.org validator.
- Confirm `index.html` default OG tags render correctly in a social debugger tool
  (e.g. Facebook Sharing Debugger, Twitter Card Validator) against the root URL.
